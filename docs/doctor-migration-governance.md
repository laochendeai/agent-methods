# Doctor / Config Repair / Migration Governance 方法

很多仓库会写 restart、release、closeout，但仍然缺一层专门处理这类问题的方法：

- 升级后为什么突然不能用了
- 配置为什么一半新一半旧
- 哪些问题只该 warning，哪些必须阻断
- 哪些能自动修，哪些必须让用户手动确认

如果没有这层，用户最终只能在报错堆里盲猜。

Claude code 里真正值得迁移的，不是某个 `doctor` 界面，而是这组原则：

> 升级和配置漂移是常态，不是事故；诊断、修复、迁移、重试必须是正式机制。

## 1. 先把 doctor 当成“健康判定 + 修复分流”系统

`doctor` 不是“打印几条日志”，它至少应该回答四个问题：

1. 现在健康吗？
2. 问题属于 warning 还是 blocking error？
3. 能自动修复吗？
4. 不能自动修时，用户下一步具体该做什么？

如果这四个问题回答不出来，说明系统还只有报错，没有真正诊断。

## 2. 诊断要区分两条维度

最容易混乱的地方，是把严重性和修复方式写成一个维度。

更好的最小模型是：

### 严重性

- `warning`
  系统还能继续运行，但存在明显漂移、退化或未来风险。
- `error`
  当前已经阻断核心功能，继续执行没有意义或风险过高。

### 修复方式

- `auto-fix`
  系统可以安全、幂等地替用户完成修复。
- `manual-fix`
  系统只能给出定位和操作建议，不应擅自修改。

所以一个问题最终应该落在下面四格之一：

- warning + auto-fix
- warning + manual-fix
- error + auto-fix
- error + manual-fix

一句话：

> warning / error 说明有多严重，auto-fix / manual-fix 说明谁来修。

## 3. doctor 应该覆盖哪些诊断面

源码里值得迁移的是，`doctor` 不只看安装状态，还看上下文和配置是否处于可用区间。

一个够用的 `doctor` 至少应覆盖：

### 安装与入口健康

- 当前安装类型
- 实际调用路径
- 是否存在多份安装
- 更新权限或安装方式是否匹配

### 配置有效性

- 配置文件是否符合 schema
- 是否存在废弃字段、冲突字段、无效字段
- 环境变量是否缺失

### 运行时上下文压力

- 规则文件是否过大
- agent 描述是否过多
- `MCP` 工具上下文是否过重
- 权限规则是否存在不可达 / 被遮蔽项

### 插件 / connector / 权限健康

- 某项能力是否被禁用
- 某个 plugin 是否加载失败
- 某个 `MCP` server 是否因为策略或状态不可用

不要把这些问题全都等到真正执行失败时才暴露。

## 4. 诊断输出必须能指导动作

一个成熟的诊断输出不只是“发现问题”，还要能指导下一步动作。

建议最小输出字段至少包含：

- `issue`
  当前问题是什么
- `severity`
  `warning` 还是 `error`
- `repair_mode`
  `auto-fix` 还是 `manual-fix`
- `source`
  来自哪个配置层、哪个模块、哪个迁移点
- `impact`
  会导致什么功能不可用或退化
- `next_step`
  现在该自动修、提示修，还是阻断退出

一句话：

> 诊断信息要让用户“知道该做什么”，而不是只知道“有点不对”。

## 5. 自动修复的判断线

不是所有问题都应该自动修。

更稳妥的判断线是：只有同时满足下面条件，才适合 `auto-fix`：

- 修改目标明确
- 不会覆盖用户仍然有效的显式意图
- 多次执行结果一致
- 出错时不会把状态推入更糟的半完成态

适合自动修的典型问题：

- 旧字段安全迁移到新字段
- 可以稳定推导的新默认值
- 无争议的路径 / 名称 / 标记位更新

不适合自动修的典型问题：

- 用户意图本身不清楚
- 需要外部凭证、业务判断或人工授权
- 修复动作可能影响多个环境
- 系统无法确认谁才是真实来源

## 6. 手动修复的判断线

以下问题更适合 `manual-fix`：

- 配置合法但含义冲突
- 多个来源都声明了更高优先级意图
- 迁移后需要用户确认行为变化
- 涉及敏感信息、权限、凭证、生产环境

此时系统不应偷偷帮用户改，而应做到：

- 给出具体出错位置
- 给出剩余旧字段或冲突字段
- 给出推荐改法
- 给出修完后如何重新验证

## 7. 迁移应该是“版本化 + 幂等”的

源码里最有价值的点，不是单个迁移函数，而是迁移被明确纳入启动流程：

- 有统一 `CURRENT_MIGRATION_VERSION`
- 同步迁移按固定顺序执行
- 只有整组同步迁移完成后，才更新迁移版本
- 非阻断异步迁移失败时，下次启动继续重试

这背后的方法是：

### 版本化

- 为同步迁移集合维护一个明确版本号
- 新增同步迁移时，版本号必须递增

### 幂等

- 迁移函数重复执行不会把状态越改越坏
- 已迁移过的项再次运行时应快速返回

### 分组

- 会影响启动一致性的迁移放同步组
- 不影响启动主路径的清理类迁移可放异步组

一句话：

> 迁移不是“脚本顺手跑一下”，而是正式版本治理的一部分。

## 8. 如何避免半迁移状态

半迁移是最常见也最烦的问题之一：新字段已经写了一半，旧字段又还没清掉，系统之后处于“好像迁了、又没全迁”的状态。

推荐原则：

### 先写新位置，再删旧位置

- 不要先删旧字段，再赌新写入一定成功
- 只有新位置确认写成功后，才清理旧字段

### 保留用户意图，而不是盲目覆盖

- 如果新位置已有明确设置，要先判断是继续保留还是合并
- 不要为了“迁移完成”把用户更晚的显式配置覆盖掉

### 迁移完成点必须明确

- 对同步迁移，只有整组成功后才 bump 版本
- 对异步迁移，失败时要允许后续重试

### doctor 必须能看见残留

- 即使迁移逻辑存在，`doctor` 仍应报告残留旧字段、冲突设置和未完成状态

## 9. 推荐的最小迁移模型

一个够用的最小模型可以长这样：

```yaml
migration:
  current_version: 3
  applied_version: 2
  sync_migrations:
    - id: move_legacy_updates_to_settings
      idempotent: true
      blocking: true
    - id: rename_model_key
      idempotent: true
      blocking: true
  async_migrations:
    - id: rebuild_cached_release_notes
      idempotent: true
      blocking: false
  retry_policy:
    async_retry_on_next_start: true
```

这个模型的重点不是字段长相，而是：

- 版本明确
- 同步 / 异步明确
- blocking / non-blocking 明确
- retry 明确

## 10. 推荐的最小 doctor 检查清单

一个仓库准备补 `doctor` 体系时，至少检查：

1. 哪些问题属于 warning，哪些属于 error？
2. 哪些问题允许 auto-fix，哪些只能 manual-fix？
3. 当前有哪些 legacy config key 或半迁移残留？
4. 迁移版本如何记录？何时 bump？
5. 失败后是阻断启动，还是记录并下次重试？
6. 输出是否能告诉用户具体下一步？

本仓库提供可直接改造的清单模板：

- `templates/project/doctor-governance-checklist.md`

## 11. 设计规则

- 严重性和修复方式分开建模
- 只有幂等、低歧义、低副作用的问题才做 auto-fix
- 同步迁移完成前不要提前 bump 版本
- 非阻断迁移失败时允许重试，不要把系统直接打残
- doctor 不是替代日志，而是给出用户可执行的修复路径
- 迁移逻辑存在，不代表可以取消诊断输出
