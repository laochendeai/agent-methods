# Hook Gate 方法

`hook` 的价值不是“让代理多跑一点东西”，而是把必须前移的门禁变成稳定契约。

如果没有 hook，很多检查会退化成：

- 这次记得跑，下次忘了跑
- 靠聊天提醒，而不是靠机制
- 同样的门禁在不同仓库里反复手写

正确用法是：

> 把重复、刚性、生命周期明确的检查前移到 hook；把需要人判断或大范围验证的事情留给 skill、规则或 CI。

## 1. 先判断什么应该做成 Hook

优先做成 hook 的检查通常有 4 个特征：

- 触发点非常稳定
- 输入边界清楚
- 结果可以直接决定“继续 / 阻止 / 报警”
- 同类检查会反复出现

典型例子：

- 每次高风险工具调用前的安全门禁
- 某类修改失败后的自动诊断或回填信息
- 会话启动或结束时的固定检查
- 任务创建时的统一元数据或记录动作

不适合做成 hook 的事情：

- 范围很大、上下文太重的完整 review
- 需要复杂人工判断的产品决策
- 本就应该在 CI 里跑的完整测试矩阵
- 一次性排障脚本

## 2. Hook、Skill、Rule、CI 的边界

### Hook

- 适合固定触发点上的刚性门禁
- 强调“什么时候触发”
- 输入通常来自工具调用、session 生命周期、任务生命周期

### Skill

- 适合完整流程
- 强调“如何把一件事做完”
- 例如 issue 闭环、回归检查、发布收尾

### Rule

- 适合长期稳定约束
- 强调“必须遵守什么”
- 例如高风险操作的默认策略、项目边界

### CI

- 适合仓库级、可重复、与本地代理无关的统一检查
- 强调“提交物是否满足统一质量线”

一句话：

> Hook 前移门禁，skill 承载流程，rule 约束行为，CI 守住交付线。

## 3. 常见 Hook 触发点

从 Claude code 源码里能看到一类清楚的事件模型：

- `PreToolUse`
- `PostToolUse`
- `PostToolUseFailure`
- `SessionStart`
- `Stop`
- `StopFailure`
- `SubagentStop`
- `TaskCreated`

把这些事件理解成 4 类就够了：

### 工具调用前

- 适合阻止高风险动作
- 适合补充上下文前置检查

### 工具调用后 / 失败后

- 适合做轻量验证、回填信息、报警
- 适合做失败后的快速诊断

### 会话启动 / 结束

- 适合运行固定 preflight
- 适合在结束时做清单、摘要、落盘、告警

### 任务 / 子代理生命周期

- 适合记录任务元数据
- 适合统一收尾与验收提醒

## 4. 四种 Hook 类型该怎么用

源码里值得迁移的不是“hook 存在”，而是 hook 被分成了四种不同职责：

### `command`

- 适合本地 shell 检查
- 例如 lint、文件存在性检查、固定格式校验
- 优点是直接、快、可复用
- 风险是命令一多就会把 hook 系统变成脚本垃圾场

适合：

- 便宜、确定、可快速返回的检查

不适合：

- 长时间运行的大型任务

### `prompt`

- 适合小范围、轻量的 LLM 判断
- 例如对输入做分类、给出轻量解释

适合：

- 需要一点智能判断，但不值得开完整子代理

不适合：

- 需要强执行和可验证证据的场景

### `http`

- 适合外部审计、日志、通知系统
- 例如发 webhook、审计记录、合规回填

适合：

- 需要把事件交给仓库外系统处理

不适合：

- 把核心交付逻辑依赖在一个不稳定外部接口上

### `agent`

- 适合独立 verifier 或更强的代理式检查
- 例如“验证单元测试是否真的跑过并通过”

适合：

- 需要比 `prompt` 更强的验证能力

不适合：

- 频率极高、成本敏感的轻量门禁

## 5. 阻塞、一次性、异步、异步唤醒怎么选

### 阻塞式 hook

适合：

- 不通过就不能继续的硬门禁
- 高风险动作前检查

不要用于：

- 成本高、易抖动、容易把主流程卡死的检查

### `once`

适合：

- 一次会话内只需要执行一次的动作
- onboarding、首次提示、一次性 preflight

### `async`

适合：

- 不必阻塞主流程的审计、记录、旁路通知

风险：

- 很容易让团队误以为“既然异步跑了，就等于已经守住质量线”

### `asyncRewake`

适合：

- 平时可以后台跑，但一旦出严重结果，必须把主线程拉回来处理

可以把它理解成：

> 正常情况不拦路，发现阻塞级故障再强制回到前台。

## 6. Hook 设计原则

### 6.1 先设计 matcher，再设计命令

不要先写命令，再想“什么时候跑”。正确顺序是：

1. 先明确触发事件
2. 再明确 matcher
3. 最后才写 hook 行为

### 6.2 把贵检查留给更高层

能用便宜 `command` 解决，就不要默认上 `agent`。

### 6.3 不要把 hook 写成第二套业务逻辑

hook 只应该做门禁、诊断、通知、元数据增强，不应该慢慢长成主流程本身。

### 6.4 保证失败是可解释的

如果 hook 拦住了流程，团队必须能知道：

- 是哪个 hook 拦的
- 为什么拦
- 应该怎么修

## 7. 最小可复用 Hook Skeleton

除了文档说明，仓库里还应该提供一个可以直接改造的骨架文件。

建议放在：

- `.claude/hooks.example.jsonc`

本仓库提供的模板示例覆盖了：

- `PreToolUse` 的阻塞式命令门禁
- `PostToolUseFailure` 的异步告警
- `Stop` 的一次性总结
- `TaskCreated` 的外部回填

## 8. 一组简单判断题

如果你在设计某个检查，先问自己：

1. 这个检查是不是有稳定触发点？
2. 如果失败，是否应该立刻影响当前流程？
3. 这个检查是每次都要跑，还是只需一次？
4. 它更像本地命令、轻量判断、外部通知，还是独立 verifier？
5. 它是不是其实更适合 skill 或 CI？

这 5 个问题答完，hook 设计通常就不会跑偏。

## 9. 适合沉淀到仓库规则里的最小约束

适合写进项目 `CLAUDE.md` 的最小规则：

- 重复、刚性的固定门禁优先考虑 hook
- 高风险工具前的硬门禁应优先前移
- 大范围验证不要全部塞进 hook
- hook 失败必须可解释、可修复

不适合沉淀进规则里的内容：

- 某次临时排障命令
- 暂时性的 webhook 地址
- 一次性实验性的 matcher 细节

本仓库对应的执行资产应该是：

- `skills/hook-gate/`
