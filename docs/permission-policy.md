# Permission Policy 方法

权限策略不是一句“高风险操作要小心”，而是一套应该被明确写出来、能解释、能诊断、能复用的规则系统。

如果权限规则只是模糊提醒，团队最终会遇到三类问题：

- 本来该拦的动作被宽泛放行
- 本来可以继续的动作被过度阻塞
- 规则彼此冲突，谁覆盖谁都说不清

正确目标不是“把所有东西都禁止”，而是：

> 让允许、拒绝、询问三种行为都有边界，让高风险模式被明确识别，让冲突规则可以被诊断。

## 1. 权限策略的三种基础行为

从源码里值得迁移的第一层，是权限行为本身被明确建模：

- `allow`
- `deny`
- `ask`

### `allow`

- 用于低风险、重复、确定性高的动作
- 例如只读检查、固定范围内的安全读操作

### `deny`

- 用于默认不应被代理直接执行的动作
- 例如明显破坏性或高外溢性的命令

### `ask`

- 用于既不能默认放行，也不该永久拒绝的动作
- 本质上是“人工确认门”

一句话：

> `allow` 给效率，`deny` 给底线，`ask` 给人类判断权。

## 2. 什么应该默认 Allow

默认允许的对象应该同时满足：

- 低风险
- 可逆或只读
- 对系统状态影响可控
- 频率高，人工每次确认没有价值

典型例子：

- 读文件
- 搜索文本
- 看日志
- 运行只读诊断命令
- 列目录、看配置结构

## 3. 什么应该默认 Ask

默认询问的对象通常具备以下特征之一：

- 会修改代码或配置
- 会影响外部环境
- 会触发网络、副作用、部署、提交
- 风险大小取决于上下文，而不是命令本身

典型例子：

- 提交代码
- 推送分支
- 修改环境配置
- 重启服务
- 安装或删除依赖

## 4. 什么应该默认 Deny

默认拒绝的对象通常是：

- 高破坏性
- 高持久化
- 高外溢性
- 很难恢复

典型例子：

- 粗暴删除
- 强推
- 无边界 shell 解释器放行
- 改生产凭据
- 数据库破坏性操作

## 5. 不要宽泛放行解释器前缀

源码里有一个很值得迁移的判断：某些“看起来是 allow 某个命令”的规则，实际上会绕过整个权限体系。

危险例子包括：

- `Bash(*)`
- `Bash(python:*)`
- `Bash(node:*)`
- `PowerShell(*)`
- `PowerShell(iex:*)`

这些规则危险的原因不是它们“语法丑”，而是：

- 它们把任意代码执行入口整体放开了
- 代理可以借解释器执行几乎任何动作
- 你失去的不是一个命令，而是整条权限边界

所以最重要的原则之一是：

> 不要把解释器、eval、shell 宽前缀当成省事的 allow 规则。

## 6. 推荐的风险分层

### LOW

- 只读、可逆、局部
- 出错影响很小

### MEDIUM

- 有副作用，但通常可恢复
- 需要结合当前上下文判断

### HIGH

- 破坏性、持久化、不可逆、外溢广
- 默认不应自动执行

这套分层的意义不是做漂亮分类，而是帮助你决定：

- LOW 更接近 `allow`
- MEDIUM 更接近 `ask`
- HIGH 更接近 `deny`

## 7. 规则冲突是常态，不是例外

成熟的权限系统不只关心“有没有规则”，还关心“规则之间会不会互相遮蔽”。

常见冲突包括：

### `deny` 遮蔽 `allow`

例如：

- 你写了一个很具体的 allow
- 但更上层有一个更宽的 deny

结果：

- 具体 allow 永远不会生效

### `ask` 遮蔽 `allow`

例如：

- 你想给某个安全前缀放行
- 但更宽的 ask 覆盖了它

结果：

- 代理还是每次都被问

### 规则源互相打架

例如：

- CLI 规则
- 项目规则
- 本地个人规则

这几层如果没有优先级说明，很容易出现“每个人都以为是别人的规则有问题”。

## 8. 怎么诊断冲突 / 遮蔽

看到权限问题时，不要只问“为什么被拦了”，要按这个顺序排查：

1. 这条命令命中了哪条规则？
2. 更宽的 `deny` 或 `ask` 是否先命中了？
3. 规则来自哪里？
4. 这条 allow 是否其实永远不可达？

诊断输出至少应能回答：

- 被哪条规则拦住
- 规则来自哪一层
- 是 `deny` 遮蔽还是 `ask` 遮蔽
- 应该删、缩、还是重写哪条规则

## 9. 推荐的最小规则写法

方法仓库不需要绑定某一套运行时实现，但应该推广这样的最小原则：

- 用最具体的 allow
- 用最明确的 deny
- 用 ask 承接需要人工确认的中间地带
- 不要写宽泛解释器 allow
- 规则越宽，越需要谨慎

### 好的风格

- 具体
- 可解释
- 易诊断

### 坏的风格

- 图省事的大通配
- 含糊的“先全开，再靠人盯”
- 同时存在互相打架的宽规则

## 10. 可复用权限规则示例

除了文档，本仓库还应该提供可以直接改造的规则片段。

本仓库提供的示例文件适合做项目第一版权限骨架：

- `templates/project/permission-policy.example.md`

它会覆盖：

- 默认 allow 的低风险动作
- 默认 ask 的中风险动作
- 默认 deny 的高风险动作
- 一组明确禁止的宽泛 allow 反例

## 11. 适合沉淀到项目规则里的最小约束

适合写进项目 `CLAUDE.md` 的最小规则：

- 默认允许低风险只读动作
- 中风险副作用动作默认 ask
- 高风险动作默认 deny
- 禁止宽泛解释器 allow
- 权限冲突必须能解释来源和遮蔽关系

不适合沉淀进项目规则里的内容：

- 某次临时排障产生的例外
- 一次性调试放行
- 只在个人机器有效的本地偏好

## 12. 对应仓库资产

本仓库对应的执行资产应该是：

- `skills/permission-policy/`

- 本文档

三者一起，才能把“权限谨慎”升级成真正可复用的权限方法。
