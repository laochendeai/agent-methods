# 采纳蓝图

## 目标

把外部项目的先进思想变成你自己的长期资产，而不是一次性的参考资料。

## 仓库边界

建议长期保持 3 层边界：

1. 业务仓库
   放产品代码、项目专属规则、运行和测试命令

2. 方法仓库
   放跨项目通用技能、交付流程、规则模板、方法文档

3. 个人层
   放你自己的偏好，不进 Git

## 为什么要单独建方法仓库

因为这些东西和业务代码不是一类资产：

- 业务仓库关注“产品能不能跑”
- 方法仓库关注“代理以后怎么做事”

把两者混在一起，业务仓库会越来越重；把方法独立出来，后续所有项目都能复用。

## 你的工程操作系统建议

### 1. 规则层

每个业务仓库保留一个 `CLAUDE.md`，只写项目事实：

- 项目目标
- 运行方式
- 测试命令
- 架构边界
- 仓库闭环流程
- 明确禁止项

### 2. 技能层

把可重复流程抽出来做 skill。你的第一批 skill 应该只覆盖高频高价值动作：

- `issue-closed-loop`
- `regression-guard`
- `runtime-restart-verify`
- `method-extract`
- `memory-promote`

### 3. 验证层

以后所有改动默认经过三类检查：

1. 影响面判断
2. 有针对性的验证
3. 关键路径 smoke

这比“改完就结束”更接近真正的工程闭环。

### 4. 隔离层

当任务复杂或容易串扰时，用分支或 worktree 隔离，而不是在一个脏工作区里混做。

## 推荐的日常工作流

1. 先读 issue 和业务仓库 `CLAUDE.md`
2. 建独立分支
3. 只做该 issue 的最小闭环
4. 跑针对性验证
5. 提交、推送、PR、合并
6. 同步并回到干净 `master`
7. 默认清理这次任务的临时分支 / worktree；如果必须保留，要写明保留原因
8. 如果这个流程以后还会重复，就把它提升成 skill 或规则

## 收尾规则默认值

- 一次性 issue 分支 / worktree 在合并后默认可清理
- 仓库收尾时应审视已合并的陈旧分支 / worktree，并清理没有保留理由的对象
- 如果某个业务仓库需要长期保留特定命名空间，例如 `backup/*`、`wip/*`，要在该业务仓库规则里显式声明
- 未合并分支默认是审查对象，不是自动清理对象

## 规则沉淀原则

只沉淀稳定、重复、长期有效的东西。

应该沉淀：

- 项目运行方式
- 数据库前提
- PR/merge 规范
- 不可触碰的架构边界
- 回归检查要求

不应该沉淀：

- 一次性 debug 结果
- 某次临时 workaround
- 个人当下情绪化偏好
- 已经过时的迁移阶段噪音

## 下一步建议

### Phase 1

- 用本仓库提供的 skills
- 让你的主项目继续用项目级 `CLAUDE.md`
- 以后新需求默认走 issue 闭环

### Phase 2

- 给主项目补 issue 模板和 PR 模板
- 补一套更明确的“定义完成”规范
- 把常见检查脚本化

### Phase 3

- 为复杂问题加轻量 worktree 流程
- 把“并行分析”和“并行实现”标准化
- 让方法仓库成为所有项目的通用底座

## 成功标准

如果这套方法真正起作用，你会看到 4 个结果：

1. 代理改完代码后，连带故障明显减少
2. 新项目启动速度更快，因为规则和技能可直接复用
3. 需求推进更稳定，因为每个 issue 都走闭环
4. 你的工程知识开始积累成资产，而不是散落在聊天记录里
