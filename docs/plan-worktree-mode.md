# Plan Mode 与 Worktree 模式

`plan mode` 和 `worktree` 不是“做事前的建议动作”，而是两种应该被明确切换的工作模式。

它们解决的是两个不同问题：

- `plan mode` 解决“复杂任务还没想清楚就开始改”的问题
- `worktree` 解决“多个任务共用一个工作区，互相污染”的问题

如果只把它们当成经验提醒，代理很容易退回到：

- 一边读代码一边顺手改文件
- 在脏工作区里并行做多个问题
- 最后很难解释这次改动为什么这样做、影响了什么、还能不能安全回收

## 1. 什么时候进入 Plan Mode

以下任一条件成立，就应该先进入 `plan mode`：

- 需求本身还不够清楚，边界没有锁定
- 预计会跨 3 个以上文件或跨多个模块
- 需要先比较多种实现方案
- 涉及架构边界、数据流、权限、迁移、发布风险
- 你已经感觉“直接开始改，八成会返工”

以下场景通常不需要单独进入 `plan mode`：

- 单文件小修复
- 明确的文案、链接、注释、格式修正
- 已有成熟模式的机械补齐
- 已经被 issue 和现有实现约束得非常清楚的微改动

## 2. Plan Mode 的允许动作

进入 `plan mode` 后，默认只做这些事：

- 读取 issue、规则文件、相关实现和相邻模块
- 搜索现有模式、相似功能、历史约束
- 记录实现方案、验证矩阵、风险点
- 必要时只编辑计划文件或设计草稿

默认不做这些事：

- 直接修改实现文件
- 一边探索一边“顺手修一点”
- 在范围没锁定前写测试、改接口、改配置

核心约束只有一句：

> 在复杂任务里，先锁定方案，再开始实现。

## 3. 什么时候退出 Plan Mode

只有在下面四件事都成立时，才应该退出 `plan mode` 并进入实施：

1. 需求范围已经清楚
2. 主要实现路径已经选定
3. 验证矩阵已经列出
4. 明确知道这次不做什么

如果做不到这四点，不应该硬切到实现阶段，而应该：

- 继续收敛方案
- 把任务拆小
- 升级为新 issue
- 明确记录阻塞点

## 4. 什么时候使用 Worktree

`worktree` 不是 branch 的替代品，而是“高隔离执行模式”。

以下场景优先考虑使用 `worktree`：

- 当前工作区已经有别的未完成工作，不想互相污染
- 同一个仓库要并行推进两个以上 issue
- 改动范围大，容易误碰其他文件
- 需要长时间保留一个实验分支，但主工作区还要继续干活
- 需要让子代理/并行实现拥有独立目录和独立运行现场

以下场景通常只用 branch 就够了：

- 当前工作区干净
- 任务很小、生命周期很短
- 不需要并行
- 不需要单独运行环境或独立目录

## 5. Branch 与 Worktree 的区别

### Branch

- 解决版本线分离
- 仍然共用当前目录
- 适合单线程、短生命周期任务

### Worktree

- 同时解决版本线分离和目录隔离
- 每个任务有独立目录
- 适合并行、长生命周期、高串扰风险任务

一句话判断：

> 只需要“版本隔离”时用 branch；需要“目录隔离”时用 worktree。

## 6. 轻量 Worktree 流程骨架

### 6.1 进入前检查

- 确认当前仓库是 Git 仓库
- 确认默认分支和目标 issue
- 判断这次任务是否真的需要目录隔离
- 确认是否存在仍需保留的本地未提交工作

### 6.2 命名

worktree 名建议稳定、可回收、可追踪：

- `issue-2-plan-worktree-mode`
- `issue-18-auth-migration`
- `hotfix-login-loop`

命名规则建议：

- 只使用字母、数字、点、下划线、短横线
- 不要使用路径穿越或含糊层级
- 名称长度保持可读

### 6.3 创建

推荐在固定目录下创建，例如：

- `.claude/worktrees/<slug>`
- 或仓库外统一目录 `<workspace>/<repo>-worktrees/<slug>`

创建后要明确三件事：

- worktree 对应哪条 branch
- 从哪个基线切出
- 是否需要共享依赖目录

### 6.4 共享目录

只有在确认安全且确实节省成本时，才共享大目录，例如：

- `node_modules`
- `.venv`
- 包管理器缓存目录

不要默认复制或共享这些内容：

- 含密钥的本地配置
- 会污染状态的运行时数据
- 不清楚是否与主工作区强耦合的临时文件

### 6.5 在 worktree 中实施

进入 worktree 后，正常走 issue 闭环：

1. 读 issue 和规则
2. 实现最小范围改动
3. 做针对性验证
4. 提交、推送、PR

### 6.6 退出与收尾

完成后默认执行：

1. PR 合并或明确不合并原因
2. 切回主分支
3. 删除临时 worktree 和临时 branch
4. 用 `repo-closeout` 做最终收尾

如果要保留 worktree，必须是刻意决定，而不是“忘了删”。

## 7. Plan Mode 与 Worktree 的配合关系

这两种模式经常一起出现，但顺序不同：

### 典型顺序

1. 先进入 `plan mode`
2. 方案和验证矩阵锁定
3. 如果任务有串扰风险，再进入 `worktree`
4. 在隔离目录里实施
5. 用 `issue-closed-loop` 做交付
6. 用 `repo-closeout` 做收尾

这比“直接切 branch 开改”更稳，因为它把：

- 设计收敛
- 目录隔离
- 实施闭环
- 最终清理

拆成了四个清楚的阶段。

## 8. 失败回退方式

如果 `plan mode` 卡住：

- 不要硬进实现
- 先缩范围
- 必要时拆 issue

如果 `worktree` 不适合当前仓库：

- 退回 branch
- 但保留 `plan mode -> issue-closed-loop -> repo-closeout` 这条主线

如果 worktree 已创建但任务中止：

- 标记保留原因
- 或在确认安全后清理
- 不要让它变成没人负责的长期噪音

## 9. 适合沉淀到仓库规则里的最小约束

可以直接写进项目 `CLAUDE.md` 的最小规则通常是：

- 非 trivial 任务先 plan，再实现
- 高串扰或并行任务优先 worktree 隔离
- worktree 必须有稳定命名和收尾动作
- 复杂任务默认带验证矩阵

不建议沉淀到规则里的内容：

- 某个仓库临时的 worktree 路径偏好
- 某次实验性的共享目录技巧
- 一次性的排障目录结构
