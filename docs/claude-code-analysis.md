# Claude Code 源码分析

分析样本：

一份外部编码代理项目的本地源码镜像，用于提炼可迁移的工程方法。

## 一句话结论

这个项目本质上不是“一个带很多 prompt 的 CLI”，而是“一个给编码代理用的运行时操作系统”。

它把代理的能力拆成了几层：

- 入口编排
- 上下文注入
- 工具注册
- 任务类型
- 技能系统
- hooks
- 权限系统
- worktree 隔离

也就是说，它把“代理怎么工作”从一次性 prompt，升级成了持续可维护的工程结构。

## 1. 入口编排，而不是单点启动

关键文件：

- `src/main.tsx`
- `src/context.ts`

可迁移思想：

- 启动时提前并行触发关键预取，不把所有成本堆到主流程后面
- 把“重逻辑 import”和“运行时初始化”分层处理
- 首屏/首轮体验优先，重活能后置就后置

对你最有价值的不是复制它的 Bun/React 细节，而是复制这个判断：

> 代理系统的响应速度，很多时候取决于入口编排，而不是模型本身。

适配到你的项目：

- 把本地启动、运行时探测、数据库状态检测、AI 配置检测视为“启动编排问题”
- 把会阻塞首轮操作的检查改成提前探测或按需探测

## 2. 上下文分层，而不是所有规则塞进一个文件

关键文件：

- `src/context.ts`
- `src/projectOnboardingState.ts`

可迁移思想：

- system context 和 user context 分离
- Git 状态是快照，不是假装实时
- `CLAUDE.md` 是 repo 规则层，不是任何信息都往里塞
- onboarding 是一个显式流程，不靠用户自己猜

这说明它很清楚地区分了四类信息：

1. 系统能力和运行时状态
2. 用户个人偏好
3. 仓库级规则
4. 会话临时信息

适配到你的项目：

- 业务仓库 `CLAUDE.md` 只存放项目长期规则
- 个人偏好不进项目仓库
- 临时调试信息不进规则层
- 新仓库模板中必须有“规则分层”说明

## 3. 工具和任务是稳定契约，不是临时拼装

关键文件：

- `src/tools.ts`
- `src/Tool.ts`
- `src/tasks.ts`
- `src/QueryEngine.ts`

可迁移思想：

- 先定义工具边界，再让模型使用工具
- 先定义任务类型，再让代理决定执行路径
- QueryEngine 负责会话生命周期，工具只负责自己的能力

这背后的核心不是“工具多”，而是：

> 代理行为必须围绕稳定契约组织，否则越用越乱。

适配到你的项目：

- 不要把“重启、验证、开 issue、PR、合并”混在自由发挥里
- 把常见交付动作变成固定流程 skill
- 每个 skill 都要定义目标、步骤、成功标准、硬规则

## 4. Skill 是能力封装单元，不是附属说明

关键文件：

- `src/skills/loadSkillsDir.ts`
- `src/skills/bundled/index.ts`
- `src/skills/bundled/verify.ts`
- `src/skills/bundled/skillify.ts`
- `src/utils/skills/skillChangeDetector.ts`

可迁移思想：

- skill 按目录组织，可以动态发现和加载
- skill 自带说明、约束、触发条件
- skill 文件变化可自动重载
- skillify 这种能力说明：流程一旦重复，就该被收编成技能

这是这个项目最值得继承的一层。

适配到你的项目：

- 建一个独立的“技能仓库”，不要把跨项目方法直接塞进业务仓库
- 任何重复三次以上的流程，优先升级成 skill
- 你的 skill 优先覆盖：
  - issue 闭环
  - 回归防护
  - 运行时重启验证
  - 外部项目方法抽取
  - 规则沉淀

## 5. Hook 是质量闸门，不是装饰品

关键文件：

- `src/utils/hooks/sessionHooks.ts`

可迁移思想：

- hook 可以是命令，也可以是内存函数
- hook 跟随 session 生命周期，而不是全局乱飞
- hook 的价值是把“必须检查的事”前移

适配到你的项目：

- 你的仓库现在最需要的 hook 思想，不是复杂系统，而是简单而刚性的门禁：
  - 改动后要有针对性验证
  - 重启类需求要验证端口和关键接口
  - review 先报风险，再报总结

## 6. 权限系统的重点不是提示框，而是风险分层

关键文件：

- `src/utils/permissions/permissionSetup.ts`

可迁移思想：

- 把高风险命令和低风险命令区别对待
- 禁止“看起来只是 Bash，实际上能执行任意代码”的宽泛放行
- 自动模式必须剥掉危险权限

适配到你的项目：

- 以后给代理的规则里，要明确哪些动作是默认允许，哪些动作必须谨慎
- 例如：
  - 默认允许：读文件、搜文本、看日志、跑只读检查
  - 谨慎：删分支、覆盖文件、批量改配置、停止不明进程、强推

## 7. Worktree 解决的是并行隔离，不是 Git 花活

关键文件：

- `src/utils/worktree.ts`

可迁移思想：

- 并行工作要有目录隔离
- worktree 名称、路径、共享依赖都要经过约束
- worktree 的存在目的是减少上下文互相污染

适配到你的项目：

- 你现在最常见的问题是“这里改了，那里又坏了”
- 根因不是模型不努力，而是缺少任务隔离和验证闭环
- 后续可以给你单独做一套轻量 worktree 流程：
  - 一个 issue 一个分支
  - 必要时一个问题一个 worktree
  - 合并后自动回到干净 `master`

## 8. 内置 skill 的写法很值得学

关键样本：

- `src/skills/bundled/simplify.ts`
- `src/skills/bundled/stuck.ts`
- `src/skills/bundled/remember.ts`

这些 skill 的共同点：

- 目标非常单一
- 步骤非常清楚
- 成功标准明确
- 不是空泛建议，而是要求模型执行固定检查

这就是为什么 skill 能长期稳定复用。

## 不应该盲目照搬的部分

以下内容不建议直接搬：

- Anthropic 自身产品相关命令
- GrowthBook/analytics/telemetry
- 特定平台的企业策略或内部通道
- 过重的 UI/CLI 复杂度
- 为超大规模代理系统准备的全部 feature flag

原因很简单：

- 这些不是你的核心竞争力
- 直接搬会带来维护负担
- 你的现阶段更需要“少而硬”的方法，而不是“大而全”的壳

## 对你最有价值的最终抽象

你真正该继承的是这 6 条：

1. 规则层独立
2. 技能化沉淀
3. 闭环交付
4. 回归防护优先
5. 权限分层
6. 必要时用隔离目录并行

所以这次落地不应是“把外部源码搬进来”，而应是：

- 建一个你自己的方法/技能仓库
- 用它长期反向约束你的业务仓库
- 让代理以后在你的项目里按这套方法做事
